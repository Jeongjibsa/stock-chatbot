# Telegram Multi-User Test Scenarios

## 1. 목적

이 문서는 실제 텔레그램 운영 환경에서 다수 사용자를 초대해 `/register`, 포트폴리오 입력, 개인화 리포트 delivery 흐름을 검증하기 위한 체크리스트다.

기본 원칙:

- `채널`은 공개 브리핑 broadcast 전용이다.
- `그룹`은 온보딩과 안내, 명령 진입점 테스트용이다.
- `DM`은 개인화 등록과 개인 리포트 delivery 전용이다.

## 2. 테스트 전제

- 봇이 아래 3개 공간에 모두 연결돼 있어야 한다.
  - 공개 `채널`
  - 테스트용 `그룹`
  - 각 사용자와의 `DM`
- 최소 3명 이상의 테스트 사용자를 준비한다.
  - `User A`
  - `User B`
  - `User C`
- 각 사용자별 mock 포트폴리오는 서로 다르게 입력한다.

예시 포트폴리오:

- `User A`
  - 삼성전자
  - SK하이닉스
- `User B`
  - 현대차
  - 현대글로비스
- `User C`
  - 에코프로
  - HMM

## 3. 채널 정책 검증

### Case C1. 채널은 공개 브리핑만 제공

절차:

1. 운영자가 채널에 공개 브리핑 또는 공개 웹 링크를 게시한다.
2. 메시지 내용을 확인한다.

기대 결과:

- 보유 종목 정보가 없어야 한다.
- 특정 사용자 이름, 포트폴리오, 평균단가, 수량이 없어야 한다.
- 공개 웹 상세 브리핑 링크는 있어도 된다.

### Case C2. 채널에서는 개인화 명령을 테스트하지 않음

절차:

1. 사용자가 채널에서 `/register` 또는 `/portfolio_add`를 시도한다.

기대 결과:

- 채널은 개인화 입력 채널이 아니라는 운영 원칙을 안내해야 한다.
- 실제 운영 기준은 `그룹 또는 DM -> DM 등록 완료 -> DM delivery` 흐름이어야 한다.

## 4. 그룹 온보딩 검증

### Case G1. 그룹에서 `/register`

절차:

1. `User A`가 그룹에서 `/register` 실행
2. `User B`도 그룹에서 `/register` 실행

기대 결과:

- 계정은 생성 또는 갱신될 수 있다.
- `preferred_delivery_chat_id`는 확정되지 않아야 한다.
- 봇은 반드시 `DM에서 다시 /register 하라`는 안내를 해야 한다.

### Case G2. 그룹에서 포트폴리오 명령 시도

절차:

1. `User A`가 그룹에서 `/portfolio_add` 실행

기대 결과:

- 아직 DM 등록이 없으면 DM 등록을 먼저 요구해야 한다.
- 개인화 데이터가 그룹 메시지로 노출되면 안 된다.

## 5. DM 등록 검증

### Case D1. DM에서 `/register`

절차:

1. `User A`, `User B`, `User C`가 각각 봇과의 DM에서 `/register` 실행

기대 결과:

- 각 사용자 계정이 등록된다.
- `preferred_delivery_chat_id`가 각 DM chat 기준으로 저장된다.
- 이후 개인화 리포트는 각 DM으로만 발송된다는 안내가 보인다.

### Case D2. 중복 `/register`

절차:

1. 이미 등록된 사용자가 DM에서 `/register`를 다시 실행

기대 결과:

- 에러 없이 재등록 또는 갱신 처리된다.
- 기존 포트폴리오가 손실되면 안 된다.
- delivery 대상은 최신 DM 기준으로 유지된다.

## 6. 개인별 종목 입력 검증

### Case P1. 사용자별 서로 다른 종목 입력

절차:

1. `User A`가 `/portfolio_add`로 삼성전자, SK하이닉스 입력
2. `User B`가 `/portfolio_add`로 현대차, 현대글로비스 입력
3. `User C`가 `/portfolio_add`로 에코프로, HMM 입력

기대 결과:

- 각 사용자별로 자기 종목만 저장된다.
- 사용자 간 종목이 섞이면 안 된다.

### Case P2. `/portfolio_list`

절차:

1. 각 사용자가 DM에서 `/portfolio_list` 실행

기대 결과:

- 본인 종목만 보여야 한다.
- 다른 사용자 종목이 한 줄이라도 보이면 실패다.

### Case P3. `/portfolio_remove`

절차:

1. `User B`가 현대글로비스 삭제
2. `User B`가 다시 `/portfolio_list` 실행

기대 결과:

- 현대글로비스만 제거돼야 한다.
- 다른 종목과 다른 사용자 데이터는 그대로여야 한다.

## 7. 시장 지표 개인화 검증

### Case M1. 사용자별 다른 지표 추가

절차:

1. `User A`는 기본 지표만 유지
2. `User B`는 `/market_add`로 구리 추가
3. `User C`는 `/market_add`로 천연가스 추가

기대 결과:

- 각 사용자의 `market_items` 결과가 다를 수 있어야 한다.
- 사용자별 지표 목록이 서로 섞이지 않아야 한다.

## 8. 리포트 delivery 검증

### Case R1. DM 개인화 발송

절차:

1. 일 배치 또는 수동 실행으로 리포트 생성
2. `User A`, `User B`, `User C`의 DM 메시지를 각각 확인

기대 결과:

- 각 DM에 본인 포트폴리오 기준 리포트만 도착해야 한다.
- `User A` 메시지에 `User B` 종목이 보이면 실패다.
- 공개 링크는 같을 수 있지만, 보유 종목 섹션은 사용자별로 달라야 한다.

### Case R2. 채널 공개 브리핑 검증

절차:

1. 같은 날짜의 채널 메시지를 확인

기대 결과:

- 시장/매크로/자금/이벤트 요약만 공개된다.
- 개인 포트폴리오/종목 기사/평단/수량은 포함되지 않는다.

### Case R3. 미등록 또는 무포트폴리오 사용자

절차:

1. `/register`는 했지만 종목이 없는 사용자를 포함해 리포트 생성

기대 결과:

- 개인 DM은 받을 수 있다.
- 다만 보유 종목 섹션은 `보유 종목 없음` 또는 동등한 안전 문구여야 한다.

## 9. 실패 케이스

### Case F1. 그룹만 등록하고 DM 등록 안 한 사용자

기대 결과:

- 개인화 발송 대상이 확정되지 않아 DM 개인 리포트 발송 대상에서 제외되거나, 명시적 재등록 안내 대상으로 처리돼야 한다.

### Case F2. 여러 사용자가 같은 시간에 등록/입력

기대 결과:

- 마지막 저장 기준이 본인 계정에만 반영돼야 한다.
- 동시성 때문에 다른 사용자 데이터가 엮이면 안 된다.

### Case F3. 공개 채널에 개인화 데이터 노출

기대 결과:

- 이 경우 즉시 실패로 판단한다.
- 운영 정책은 항상 `채널=공개`, `DM=개인화`다.

## 10. 운영 제안

- 개인화 리포트는 무조건 `DM delivery`로 고정하는 편이 좋다.
- `채널`은 공개 브리핑과 공개 웹 링크만 게시한다.
- `그룹`은 사용자 온보딩 안내와 `/register` 유도만 담당한다.
- 장기적으로는 아래 3계층이 가장 안전하다.
  - `채널`: public
  - `그룹`: onboarding/support
  - `DM`: personalized delivery
